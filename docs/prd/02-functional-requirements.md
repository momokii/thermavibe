## Module 1: Kiosk Flow (FR-KIOSK)

The kiosk operates as a deterministic state machine with six states. Every user session follows this sequence without deviation. The state machine is managed by the backend and reflected in the frontend UI in real time.

### FR-KIOSK-001: Idle State (Attract Loop)

**Description:** When no session is active, the kiosk displays a continuously looping attract screen designed to draw attention and invite interaction. This screen displays animated visuals, the product branding, and a clear call-to-action instructing the user to touch the screen to begin.

**Acceptance Criteria:**
- The attract loop plays continuously with no user intervention required.
- A single touch anywhere on the screen transitions the kiosk from Idle to Payment (or Capture, if payment is disabled).
- The attract loop resumes automatically after session completion and the Reset state timeout expires.
- The attract loop content is configurable by the operator (images, video, text) through the admin dashboard.
- The attract loop runs at a minimum of 30 FPS with no visible stuttering on the target hardware.
- Audio is optional and configurable; the attract loop must function correctly with audio muted.

**Priority:** P0

---

### FR-KIOSK-002: Payment State

**Description:** When payment is enabled, the kiosk transitions from Idle to Payment upon touch. The Payment state displays a QRIS QR code generated by the configured payment provider (Midtrans or Xendit). The user scans the QR code with their mobile payment app to complete the transaction.

**Acceptance Criteria:**
- A QRIS-compliant QR code is displayed prominently on screen within 2 seconds of entering the Payment state.
- The QR code includes the correct transaction amount as configured by the operator.
- The kiosk listens for a webhook callback from the payment provider confirming successful payment.
- Upon successful payment confirmation, the kiosk transitions to the Capture state within 1 second.
- If the payment provider does not confirm payment within the configured timeout (default: 120 seconds), the kiosk transitions back to the Idle state and displays a "session expired" message for 3 seconds.
- The QR code is regenerated for each new session to prevent reuse.
- If payment is disabled by the operator, the kiosk skips the Payment state entirely and transitions directly from Idle to Capture.

**Priority:** P0

---

### FR-KIOSK-003: Capture State

**Description:** The kiosk activates the webcam and displays a live camera preview. After a configurable countdown (default: 3 seconds), the system captures a still photo. The user sees themselves on screen with a countdown overlay and visual guidance.

**Acceptance Criteria:**
- The live camera feed appears on screen within 500ms of entering the Capture state.
- A visible countdown timer (defaulting to 3 seconds, configurable from 1-10 seconds) is displayed as an overlay on the camera feed.
- The photo is captured automatically at the end of the countdown; no user tap is required.
- The captured image is displayed for 1-2 seconds as a "preview" to confirm capture before transitioning to the Processing state.
- The captured image resolution meets the minimum required for print quality (at least 640x480 pixels; recommended 1280x720).
- If the webcam is not detected or fails to initialize, the kiosk displays a clear error message and transitions to the Reset state.
- The camera feed is not mirrored (or is mirrored based on operator configuration) consistently between preview and capture.

**Priority:** P0

---

### FR-KIOSK-004: Processing State (AI Analysis)

**Description:** The captured photo is compressed and sent to the configured AI provider for analysis. The AI provider generates a "vibe reading" -- a personalized, witty personality analysis based on the user's photo. The kiosk displays a loading/processing animation during this phase.

**Acceptance Criteria:**
- The processing state displays a visually engaging loading animation to keep the user's attention.
- The captured image is compressed to the configured target size (default: under 500KB) before transmission to the AI provider.
- The system prompt, as configured by the operator, is sent along with the image to the AI provider.
- The AI response is received, parsed, and validated within the configured timeout (default: 30 seconds).
- If the AI provider returns an invalid, empty, or error response, the system attempts a retry with the fallback provider (if configured) or uses a pre-configured fallback template.
- Upon successful AI response, the kiosk transitions to the Reveal/Print state.
- If all AI attempts fail within the timeout window, the kiosk transitions to the Reset state and displays an apology message.
- The processing state does not expose any technical details (API errors, provider names, response codes) to the end-user.

**Priority:** P0

---

### FR-KIOSK-005: Reveal and Print State

**Description:** The AI-generated vibe reading is displayed on screen in a visually formatted layout alongside the captured photo. Simultaneously, the system sends a print job to the thermal printer containing a dithered version of the photo and the AI text formatted for the receipt width.

**Acceptance Criteria:**
- The vibe reading text is displayed on screen in a legible, formatted layout (minimum 24pt equivalent font size).
- The display includes the captured photo alongside the text.
- The thermal printer receives the print job within 2 seconds of entering the Reveal state.
- The printed receipt contains: (a) a dithered halftone rendering of the captured photo sized to the receipt width, (b) the AI-generated vibe reading text formatted to fit the receipt width with appropriate line wrapping, and (c) optional branding text configured by the operator.
- The printer executes a paper cut command after printing the receipt content.
- If the printer is offline or fails to respond, the kiosk displays the vibe reading on screen (so the user can photograph it) and shows a message indicating the print could not be produced.
- The Reveal state remains active for a configurable duration (default: 15 seconds) to give the user time to read the screen and collect their receipt before transitioning to Reset.
- The user may tap the screen during the Reveal state to skip directly to Reset.

**Priority:** P0

---

### FR-KIOSK-006: Reset State

**Description:** After the Reveal/Print state completes (either by timeout or user tap), the kiosk enters the Reset state. All session data is purged from memory, the camera is released, and the kiosk transitions back to the Idle state.

**Acceptance Criteria:**
- All session data (captured photo, AI response, payment transaction ID) is deleted from volatile memory during the Reset state.
- No user photo data is written to persistent storage at any point during the session lifecycle.
- The camera stream is stopped and the hardware resource is released.
- The Reset state completes within 2 seconds and transitions to the Idle state.
- If a new user touches the screen during the Reset state, the touch is ignored until the Idle state is fully active.
- The state machine guarantees that no state is skipped and no state is entered out of order under normal operation.

**Priority:** P0

## Module 2: Admin Dashboard (FR-ADMIN)

The admin dashboard is a web-based interface accessible from any device on the local network. It is protected by a PIN code and provides operators with full control over the kiosk configuration, hardware management, and analytics.

### FR-ADMIN-001: Hardware Setup and Status

**Description:** The admin dashboard displays the current status of all connected hardware components (webcam, thermal printer) and provides tools for testing and troubleshooting.

**Acceptance Criteria:**
- The dashboard shows real-time connection status for the webcam (connected/disconnected, device name, supported resolutions).
- The dashboard shows real-time connection status for the thermal printer (connected/disconnected, device name, paper status if available via ESC/POS).
- A "Test Camera" button captures a test photo and displays it in the dashboard without affecting the kiosk's current state.
- A "Test Print" button sends a test print job (containing a test pattern and text) to the thermal printer.
- If a hardware device is disconnected, the dashboard displays a warning banner with suggested troubleshooting steps.
- The hardware status is refreshed automatically every 10 seconds or can be manually refreshed.
- The dashboard lists all detected USB devices of the relevant classes (video, printer) to assist with hardware identification.

**Priority:** P0

---

### FR-ADMIN-002: AI Provider Configuration

**Description:** The admin dashboard allows the operator to configure the AI provider, including selecting the provider, entering API keys, choosing the model, and setting the system prompt.

**Acceptance Criteria:**
- The operator can select the AI provider from a dropdown: OpenAI, Anthropic, Google, or Ollama (local).
- The operator can enter and save the API key for the selected provider. API keys are stored encrypted at rest and are never displayed in full after initial entry (masked with asterisks).
- The operator can select the specific model to use (e.g., GPT-4o, Claude 3.5 Sonnet, Gemini Pro, etc.) from a list of supported models for the selected provider.
- The operator can compose and save a custom system prompt in a multi-line text editor. The system prompt defines the personality, tone, and format of the AI vibe reading.
- A "Preview" button sends a test image to the configured provider with the current system prompt and displays the resulting AI response in the dashboard, allowing the operator to iterate on the prompt before deploying.
- The operator can configure a fallback provider (different from the primary) that is used if the primary provider fails or times out.
- Configuration changes take effect on the next session; they do not interrupt an active session.

**Priority:** P0

---

### FR-ADMIN-003: Payment Configuration

**Description:** The admin dashboard allows the operator to configure payment settings, including enabling/disabling payment, setting the price, selecting the payment provider, and entering payment gateway credentials.

**Acceptance Criteria:**
- The operator can toggle payment on or off with a single switch. When payment is off, the kiosk skips the Payment state.
- The operator can set the session price in Indonesian Rupiah (IDR) with a minimum of IDR 1 and a maximum of IDR 1,000,000.
- The operator can select the payment provider (Midtrans or Xendit) and enter the corresponding server key and merchant ID.
- Payment credentials are stored encrypted at rest and never exposed to the frontend.
- A "Test Payment" button generates a test QRIS QR code and displays it in the dashboard, along with a simulated payment confirmation for verification.
- The operator can configure the payment timeout (default: 120 seconds; configurable from 30 to 300 seconds).
- The operator can configure the currency symbol and display format.

**Priority:** P1

---

### FR-ADMIN-004: AI Prompt Customization

**Description:** The admin dashboard provides a dedicated interface for customizing the AI system prompt, including pre-built templates and a live preview capability.

**Acceptance Criteria:**
- The dashboard includes a library of pre-built system prompt templates (e.g., "Coffee Shop Vibe," "Music Festival Energy," "Minimalist Elegance," "Playful and Fun") that the operator can select as a starting point.
- The operator can modify any pre-built template or write a completely custom system prompt from scratch.
- The system prompt editor supports a character count display and a recommended length indicator (200-1000 characters).
- A "Save and Test" button sends a sample image to the AI provider with the current prompt and displays the result, enabling rapid iteration.
- Previously saved prompts are stored in a version history, and the operator can revert to any previous version.
- The prompt editor includes placeholder variables (e.g., `{{venue_name}}`, `{{price}}`) that are automatically substituted at runtime with the operator's configured values.
- The current active prompt is displayed in read-only form on the main dashboard overview page.

**Priority:** P1

---

### FR-ADMIN-005: Analytics and Reporting

**Description:** The admin dashboard provides operators with analytics on kiosk usage, revenue, session completion rates, and hardware health over configurable time ranges.

**Acceptance Criteria:**
- The dashboard displays a summary of key metrics for the current day: total sessions, completed sessions, failed sessions, total revenue, average session duration.
- The operator can select a date range (today, last 7 days, last 30 days, custom range) for all analytics views.
- A line chart shows sessions per day over the selected range, with separate lines for completed and failed sessions.
- A bar chart shows revenue per day over the selected range.
- A pie chart shows session outcome distribution (completed, payment timeout, AI failure, printer failure, user abandonment).
- The dashboard calculates and displays the session completion rate (completed sessions / total started sessions) as a percentage.
- The dashboard calculates and displays the average end-to-end session duration (touch to print).
- Analytics data is stored in the PostgreSQL database and persists across system restarts.
- All analytics views are exportable as CSV for external analysis.

**Priority:** P1

---

### FR-ADMIN-006: Session Logs

**Description:** The admin dashboard maintains a searchable, filterable log of all kiosk sessions with sufficient detail for troubleshooting and business analysis.

**Acceptance Criteria:**
- Every session is logged with: session ID (UUID), start timestamp, end timestamp, outcome (completed/failed/abandoned), failure reason (if applicable), duration of each state, payment amount and transaction ID (if applicable), AI provider used and response time, and printer status.
- The session log is displayed as a paginated table with configurable page size (10, 25, 50, 100 rows).
- The operator can filter sessions by: date range, outcome (completed, failed, abandoned), failure reason, and payment status.
- The operator can search sessions by session ID.
- Each session row can be expanded to show detailed state-by-state timing information.
- Session logs are retained for a configurable period (default: 90 days) and automatically purged after the retention period expires.
- The session log does not contain any user photo data or personally identifiable information.

**Priority:** P1

## Module 3: Camera (FR-CAMERA)

The camera module handles all interaction with the USB webcam, including device detection, stream management, and photo capture. It is implemented using OpenCV and is abstracted behind a Python interface.

### FR-CAMERA-001: Camera Preview Stream

**Description:** The camera module provides a live video preview stream that is displayed on the kiosk screen during the Capture state and optionally during the Idle state.

**Acceptance Criteria:**
- The camera preview is delivered as an MJPEG stream served by the backend at a configurable endpoint (default: `/api/camera/stream`).
- The preview stream is displayed in the frontend using a standard `<img>` tag pointing to the MJPEG endpoint.
- The preview latency (time from real-world event to display on screen) does not exceed 200 milliseconds on the target hardware.
- The preview resolution is configurable independently of the capture resolution (default: 640x480).
- The preview stream is started when the kiosk enters the Capture state and stopped when the kiosk exits the Capture state to free hardware resources.
- If the camera is already in use by another process, the module reports a clear error and does not crash.

**Priority:** P0

---

### FR-CAMERA-002: Photo Capture

**Description:** The camera module captures a high-resolution still image from the webcam at the moment the countdown reaches zero.

**Acceptance Criteria:**
- The capture resolution is configurable (default: 1280x720; minimum: 640x480; maximum: 1920x1080).
- The captured image is returned as a JPEG-encoded byte array with configurable quality (default: 85%).
- The capture operation completes within 100 milliseconds from the trigger command.
- The captured image is stored in volatile memory only (Python variable) and is never written to the filesystem.
- If the camera fails to capture (hardware error, buffer overrun), the module returns a structured error that the kiosk state machine can handle (transition to Reset state with error message).
- The capture includes accurate EXIF metadata for orientation if available from the camera hardware.

**Priority:** P0

---

### FR-CAMERA-003: Camera Resolution Configuration

**Description:** The operator can configure the camera preview and capture resolutions through the admin dashboard. The system validates that the requested resolution is supported by the detected hardware before applying the setting.

**Acceptance Criteria:**
- The admin dashboard displays all resolutions supported by the detected webcam.
- The operator can select a preview resolution and a capture resolution independently.
- If the operator selects a resolution not supported by the hardware, the dashboard displays a validation error and does not apply the change.
- Resolution changes take effect on the next session; they do not interrupt an active session.
- The system falls back to the camera's default resolution if no configuration is provided.

**Priority:** P1

---

### FR-CAMERA-004: Hardware Detection and Enumeration

**Description:** On system startup and on demand, the camera module scans for connected UVC-compliant USB webcams and reports their capabilities.

**Acceptance Criteria:**
- On startup, the system automatically detects all connected UVC webcams and logs their device paths, names, and supported resolutions.
- If multiple webcams are connected, the operator can select which one to use through the admin dashboard.
- If no webcam is detected, the system logs a warning and the admin dashboard displays a prominent error banner.
- If the configured webcam is disconnected during operation, the system detects the disconnection within 5 seconds and updates the admin dashboard.
- When a previously disconnected webcam is reconnected, the system detects the reconnection and automatically resumes camera functionality.
- The detection mechanism uses the device's USB vendor and product IDs to distinguish between different cameras across reboots.

**Priority:** P0

---

### FR-CAMERA-005: MJPEG Stream Management

**Description:** The backend serves the camera preview as an MJPEG stream with proper lifecycle management to prevent resource leaks.

**Acceptance Criteria:**
- The MJPEG stream endpoint starts serving frames only when requested by a client and stops when the client disconnects.
- The stream supports a single client at a time (the kiosk frontend). If a second client connects, the first client's stream is terminated.
- The stream includes proper HTTP headers (Content-Type: multipart/x-mixed-replace).
- The stream framerate is configurable (default: 15 FPS; range: 5-30 FPS).
- If the camera becomes unavailable while the stream is active, the stream is terminated gracefully with a clean HTTP response rather than a connection reset.
- The stream does not accumulate frames in a buffer; only the most recent frame is served to minimize latency.

**Priority:** P1

---

## Module 4: AI Engine (FR-AI)

The AI engine handles communication with AI providers, including image compression, request dispatch, response parsing, and fallback management. It is designed to be provider-agnostic through an abstracted interface.

### FR-AI-001: Image Compression and Preprocessing

**Description:** Before sending the captured photo to an AI provider, the system compresses and preprocesses the image to meet the provider's requirements and to minimize latency and cost.

**Acceptance Criteria:**
- The captured image is resized to a maximum dimension of 1024 pixels on the longest side while preserving aspect ratio.
- The resized image is encoded as JPEG with configurable quality (default: 80%; minimum: 50%; maximum: 95%).
- The compressed image size does not exceed the configured maximum (default: 500KB). If the image exceeds the maximum after initial compression, quality is reduced incrementally until the size target is met.
- The preprocessing pipeline strips all EXIF metadata (including GPS coordinates) from the image before transmission to protect user privacy.
- The preprocessing operation completes within 500 milliseconds on the target hardware.
- The system logs the compressed image size for each session for operator monitoring.

**Priority:** P0

---

### FR-AI-002: Multi-Provider Dispatch

**Description:** The AI engine supports multiple providers (OpenAI, Anthropic, Google, Ollama) through a unified interface. The operator selects the primary provider, and the system dispatches requests accordingly.

**Acceptance Criteria:**
- Each supported provider implements a common interface with methods: `analyze_image(image_bytes, system_prompt, model)` returning a structured response object.
- The provider selection is configurable via the admin dashboard and stored in the database.
- Each provider adapter handles the specific API format, authentication header, and response schema for its respective service.
- The dispatch layer adds appropriate logging (request timestamp, provider name, model, response timestamp, response status) without logging the image data or API key.
- New providers can be added by implementing the provider interface without modifying the dispatch layer (Open/Closed Principle).
- The dispatch layer measures and reports the response time for each provider call.

**Priority:** P0

---

### FR-AI-003: System Prompt Configuration

**Description:** The operator defines a system prompt that shapes the AI's personality, tone, and output format for vibe readings. The prompt is stored in the database and applied to every AI request.

**Acceptance Criteria:**
- The system prompt is stored as a text field in the database with a maximum length of 5000 characters.
- The system prompt supports template variables (e.g., `{{venue_name}}`, `{{date}}`, `{{price_currency}}`) that are substituted at runtime.
- If the system prompt is empty or not configured, the system uses a sensible default prompt that produces a fun, concise personality reading.
- The system prompt is included in every AI request as the system/context message according to the provider's API specification.
- Changes to the system prompt are applied to the next session and do not affect sessions already in progress.
- The admin dashboard shows a character count and provides guidance on effective prompt structure.

**Priority:** P0

---

### FR-AI-004: Response Parsing and Validation

**Description:** The AI engine parses the provider's response, extracts the vibe reading text, and validates that the response meets quality criteria before proceeding to the Reveal/Print state.

**Acceptance Criteria:**
- The response parser extracts the text content from the provider's response object, handling the specific format of each provider (OpenAI's `choices[0].message.content`, Anthropic's `content[0].text`, etc.).
- The extracted text is validated against minimum length (at least 20 characters) and maximum length (at most 500 characters) constraints.
- If the response contains JSON or markdown formatting artifacts, the parser strips them to produce clean plain text.
- If the response fails validation (too short, too long, empty, or contains only whitespace), the system treats it as a failed response and triggers the fallback flow.
- The validated text is stored in the session record for display and printing.
- The parser handles edge cases: provider returns a refusal, provider returns an empty content block, provider returns non-text content (e.g., a tool call).

**Priority:** P0

---

### FR-AI-005: Fallback Handling

**Description:** When the primary AI provider fails (timeout, error response, invalid output), the system automatically attempts the request with a configured fallback provider before giving up.

**Acceptance Criteria:**
- The operator can configure a fallback provider in the admin dashboard, which may be different from the primary provider.
- If the primary provider fails, the system sends the same image and system prompt to the fallback provider without requiring user interaction.
- The system makes a maximum of 1 fallback attempt per session (2 total attempts: primary + fallback).
- If both the primary and fallback providers fail, the system uses a pre-configured fallback template -- a set of pre-written, generic-but-fun vibe readings that are randomly selected.
- The fallback template ensures that the user always receives a printed receipt even if all AI providers are unavailable.
- The fallback flow is logged in the session record with the failure reasons for each provider attempt.
- The operator can customize the fallback templates through the admin dashboard.

**Priority:** P1

---

### FR-AI-006: Timeout Management

**Description:** Each AI provider request is subject to a configurable timeout. If the provider does not respond within the timeout, the request is cancelled and the fallback flow is triggered.

**Acceptance Criteria:**
- The AI request timeout is configurable per provider (default: 20 seconds for cloud providers; default: 60 seconds for Ollama).
- The timeout applies to the entire HTTP request, including connection establishment, request transmission, and response reception.
- When a timeout occurs, the HTTP connection is properly closed and any in-flight request is cancelled.
- The timeout value is displayed in the admin dashboard alongside the configured provider settings.
- The system distinguishes between timeout errors and other errors (authentication failure, rate limiting, server error) in its logging and fallback logic.
- The cumulative timeout for all AI attempts (primary + fallback) must not exceed the Processing state timeout (default: 30 seconds).

**Priority:** P0

## Module 5: Thermal Printer (FR-PRINT)

The printer module handles all interaction with the ESC/POS-compatible thermal printer, including image dithering, text formatting, print job construction, and hardware communication. It is implemented using the python-escpos library.

### FR-PRINT-001: Image Dithering

**Description:** The captured photo is converted to a dithered halftone image suitable for printing on a monochrome thermal printer. The dithering algorithm produces a visually appealing, high-contrast rendering that mimics the aesthetic of newspaper halftone prints.

**Acceptance Criteria:**
- The dithering algorithm supports Floyd-Steinberg and Atkinson dithering methods, selectable by the operator.
- The dithered output is a 1-bit monochrome bitmap (black and white only) matching the printer's native resolution.
- The image is scaled to fit the receipt width (58mm paper: 384 pixels; 80mm paper: 576 pixels) while preserving aspect ratio.
- The dithering operation completes within 1 second on the target hardware for a 1280x720 source image.
- The operator can configure the dithering method and the image position on the receipt (centered, left-aligned, right-aligned) through the admin dashboard.
- The dithered image quality is sufficient for the subject's face to be clearly recognizable on the printed receipt.

**Priority:** P0

---

### FR-PRINT-002: Text Formatting for Receipt

**Description:** The AI-generated vibe reading text is formatted to fit within the thermal printer's character width with appropriate line wrapping, alignment, and emphasis.

**Acceptance Criteria:**
- The text is formatted for the printer's character width (58mm paper: 32 characters per line; 80mm paper: 48 characters per line).
- Long text is automatically word-wrapped at the appropriate column width without breaking words mid-syllable where possible.
- The text is rendered in a readable font size (the printer's default font or a configured alternative).
- The text formatting supports bold and double-height styles for headings or emphasis.
- The text is center-aligned by default; the operator can configure alignment (left, center, right) through the admin dashboard.
- If the text exceeds the maximum printable length (approximately 400 characters for 58mm paper), it is truncated with an ellipsis and the full text is displayed on screen instead.
- The formatted text is previewed in the admin dashboard before printing.

**Priority:** P0

---

### FR-PRINT-003: ESC/POS Print Job Construction

**Description:** The printer module constructs a complete ESC/POS print job containing the dithered image, the formatted text, and optional branding elements, then sends it to the thermal printer.

**Acceptance Criteria:**
- The print job is constructed as a sequence of ESC/POS commands in the correct order: initialize printer, set alignment, print image, feed lines, print text, feed lines, print footer (if configured), cut paper.
- The print job includes an initialization command at the beginning to reset the printer to a known state.
- The print job includes line feeds between the image and text sections for visual separation.
- The print job includes a paper cut command (`GS V 66 0`) at the end.
- The operator can configure a footer line (e.g., venue name, social media handle, hashtag) that is printed below the AI text.
- The print job is sent to the printer as a single batch to prevent interleaving with other print jobs.
- The total print time for a standard session receipt (image + text + footer) does not exceed 8 seconds.

**Priority:** P0

---

### FR-PRINT-004: Paper Cut Command

**Description:** After printing the receipt content, the printer module sends a paper cut command to the thermal printer to separate the current receipt from the paper roll.

**Acceptance Criteria:**
- The paper cut command is sent immediately after the last content element of the receipt.
- The paper cut command uses the standard ESC/POS sequence (`GS V 66 0` for full cut or `GS V 1` for partial cut).
- The operator can configure full cut or partial cut through the admin dashboard.
- If the printer does not support the paper cut command, the module handles the error gracefully and does not crash (the receipt can be torn manually).
- The paper cut occurs after sufficient line feeds to ensure the cut line is below the printed content.

**Priority:** P1

---

### FR-PRINT-005: Test Print Functionality

**Description:** The printer module provides a test print function that produces a diagnostic receipt without requiring a full kiosk session. This is used for initial setup, troubleshooting, and paper alignment verification.

**Acceptance Criteria:**
- A test print can be triggered from the admin dashboard via a "Test Print" button.
- The test print receipt contains: (a) a header with "VibePrint OS Test Print" and the current timestamp, (b) a dithered test pattern (gradient or checkerboard) to verify image printing, (c) sample text in normal, bold, and double-height fonts, (d) a footer line, and (e) a paper cut.
- The test print does not create a session record in the database.
- The test print does not require the AI provider to be configured or available.
- The test print uses the same print job construction pipeline as production prints, ensuring that a successful test print guarantees production print correctness.
- If the test print fails, the admin dashboard displays a descriptive error message (e.g., "Printer not connected," "Paper out," "Communication error").

**Priority:** P1

---

## Module 6: Payment (FR-PAYMENT)

The payment module handles QRIS payment integration through supported providers (Midtrans and Xendit). It is designed to be toggle-able and includes a mock provider for testing and demonstration purposes.

### FR-PAYMENT-001: QRIS QR Code Generation

**Description:** When a session enters the Payment state, the payment module generates a QRIS-compliant QR code for the configured transaction amount and displays it on the kiosk screen.

**Acceptance Criteria:**
- The payment module calls the configured provider's API to create a payment transaction with the session amount and a unique order ID (tied to the session ID).
- The provider returns a QRIS QR code string or URL, which the module encodes as a PNG image.
- The QR code image is at least 300x300 pixels for reliable scanning by mobile devices.
- The QR code is displayed on the kiosk screen within 2 seconds of entering the Payment state.
- The session amount displayed alongside the QR code matches the amount sent to the payment provider.
- The order ID is unique per session and includes a timestamp to prevent collisions.
- The QR code is valid for scanning by any Indonesian mobile payment app that supports QRIS (GoPay, OVO, DANA, ShopeePay, bank apps, etc.).

**Priority:** P0

---

### FR-PAYMENT-002: Payment Confirmation Webhook

**Description:** After the QR code is generated, the payment module listens for a webhook callback from the payment provider confirming that the user has completed the payment.

**Acceptance Criteria:**
- The backend exposes a webhook endpoint (e.g., `/api/payment/webhook`) that receives POST callbacks from the payment provider.
- The webhook handler validates the webhook signature (as specified by the provider's documentation) to ensure the callback is authentic.
- Upon receiving a successful payment confirmation, the handler looks up the corresponding session and triggers the state transition from Payment to Capture.
- The webhook handler is idempotent: receiving the same webhook multiple times for the same transaction does not cause duplicate state transitions or errors.
- The webhook endpoint responds with HTTP 200 within 500 milliseconds to prevent the provider from retrying.
- Failed webhook validations (invalid signature, unknown transaction ID) are logged but do not cause system errors.
- The webhook endpoint is secured to prevent unauthorized access (only the payment provider's IP range or authenticated requests are accepted).

**Priority:** P0

---

### FR-PAYMENT-003: Payment Timeout Handling

**Description:** If the user does not complete the payment within the configured timeout, the session is cancelled and the kiosk returns to the Idle state.

**Acceptance Criteria:**
- The payment timeout is configurable (default: 120 seconds; range: 30 to 300 seconds).
- A visible countdown timer is displayed on the kiosk screen alongside the QR code, showing the remaining time to complete payment.
- When the timeout expires, the kiosk cancels the payment transaction with the provider (via API call) to prevent late payments from being charged.
- The kiosk transitions to the Idle state and displays a "Session expired. Please try again." message for 3 seconds before resuming the attract loop.
- If a payment confirmation arrives after the timeout has expired (race condition), the system logs the event and refunds the payment automatically via the provider's refund API.
- The timeout value is displayed in the admin dashboard and is configurable without restarting the service.

**Priority:** P0

---

### FR-PAYMENT-004: Payment Toggle (Enable/Disable)

**Description:** The operator can enable or disable payment through the admin dashboard. When payment is disabled, the kiosk skips the Payment state entirely and allows free sessions.

**Acceptance Criteria:**
- The payment toggle is a single switch in the admin dashboard with immediate effect (applied on the next session).
- When payment is disabled, the kiosk transitions directly from Idle to Capture upon user touch, with no QR code displayed.
- When payment is disabled, the webhook endpoint remains active but ignores incoming callbacks (to handle edge cases where a payment was initiated before the toggle was switched).
- The current payment state (enabled/disabled) is clearly visible on the admin dashboard main page.
- Toggling payment does not require restarting the Docker containers or the backend service.
- The kiosk screen displays appropriate messaging: when payment is enabled, the price is shown; when payment is disabled, "Free" or "Complimentary" is shown.

**Priority:** P1

### FR-PAYMENT-005: Mock Payment Provider

**Description:** A mock payment provider is available for development, testing, and demonstration purposes. It simulates the full payment flow without connecting to a real payment gateway.

**Acceptance Criteria:**

## Module 7: Failsafe and Error Recovery (FR-FAILSAFE)

The failsafe module ensures that the kiosk can recover gracefully from errors, connectivity issues, and unexpected conditions without manual intervention. It is critical for unattended operation.

### FR-FAILSAFE-001: Connectivity Monitoring

**Description:** The system continuously monitors its connectivity to external services (AI provider API, payment provider API) and takes appropriate action when connectivity is lost or degraded.

**Acceptance Criteria:**
- The system performs a health check to the configured AI provider's API endpoint every 30 seconds.
- The system performs a health check to the configured payment provider's API endpoint every 30 seconds (when payment is enabled).
- The health check results are displayed in the admin dashboard (green/yellow/red status indicators for each external service).
- When an external service becomes unreachable, the system logs the event and updates the dashboard but does not crash or restart.
- When connectivity is restored, the system logs the recovery and resumes normal operation automatically.
- If the AI provider is unreachable when a session reaches the Processing state, the system immediately triggers the fallback flow (fallback provider or fallback template) without waiting for a timeout.
- The kiosk screen does not display any connectivity status information to the end-user.

**Priority:** P0

---

### FR-FAILSAFE-002: Timeout Limits and Enforcement

**Description:** Each state in the kiosk state machine has a maximum duration. If a state exceeds its timeout, the system transitions to the Reset state and logs the timeout event.

**Acceptance Criteria:**
- Each state has a configurable maximum timeout: Idle (unlimited), Payment (configurable, default 120s), Capture (30 seconds), Processing (30 seconds), Reveal/Print (30 seconds), Reset (5 seconds).
- When a state timeout is reached, the system cancels any in-progress operations (AI request, print job), logs the timeout with the state name and elapsed time, and transitions to the Reset state.
- State timeouts are enforced by a dedicated timer mechanism (not dependent on the main event loop) to ensure they fire even if the system is under load.
- The operator can configure state timeouts through the admin dashboard or via environment variables.
- Timeout events are counted in the analytics dashboard under "Failed Sessions" with the failure reason "State Timeout."
- The end-user sees a polite "Something went wrong. Please try again." message if a timeout occurs, with no technical details exposed.

**Priority:** P0

---

### FR-FAILSAFE-003: Fallback Templates

**Description:** When the AI engine cannot produce a valid response (all providers fail or time out), the system uses a set of pre-configured fallback templates to ensure the user still receives a receipt.

**Acceptance Criteria:**
- A minimum of 10 unique fallback templates are included with the default installation.
- Fallback templates are stored in the database and are editable by the operator through the admin dashboard.
- When a fallback template is used, it is randomly selected from the available templates to provide variety across sessions.
- The fallback template includes the same formatting and structure as a real AI response (greeting, personality analysis, closing).
- The session record clearly indicates that a fallback template was used (not a live AI response).
- The analytics dashboard tracks the rate of fallback template usage as a percentage of total sessions, serving as an early warning indicator of AI provider issues.
- Fallback templates do not reference the user's appearance (since no AI analysis was performed); they are general, fun personality readings.

**Priority:** P1

---

### FR-FAILSAFE-004: Privacy and Data Clearing

**Description:** The system implements strict data lifecycle management to ensure that user photos and session data are not retained beyond the session lifecycle, protecting user privacy.

**Acceptance Criteria:**
- Captured photos are held in volatile memory (Python process memory) only and are never written to the filesystem, database, or any external storage.
- After the Reset state completes, all references to the captured photo in memory are explicitly deleted (set to `None` / garbage collected).
- The AI provider API is called over HTTPS; the image is transmitted in the request body and is not logged by the application.
- Session metadata (timestamps, outcome, provider used, response time) is stored in the PostgreSQL database but does not include the photo or any personally identifiable information.
- The admin dashboard does not display, expose, or allow retrieval of user photos.
- Session metadata is automatically purged from the database after the configured retention period (default: 90 days).
- A manual "Purge All Data" button in the admin dashboard immediately deletes all session records from the database (with a confirmation dialog).
- The system does not transmit user photos to any service other than the configured AI provider.

**Priority:** P0

---

### FR-FAILSAFE-005: Error Recovery and Automatic Restart

**Description:** When a component of the system fails (backend crash, database disconnection, printer error), the system recovers automatically without requiring manual intervention.

**Acceptance Criteria:**
- Docker Compose is configured with `restart: unless-stopped` for all services, ensuring automatic restart on crash.
- If the FastAPI backend crashes, Docker restarts it within 5 seconds, and the kiosk returns to the Idle state after the backend is healthy.
- If the PostgreSQL database connection is lost, the backend enters a degraded mode: sessions can still run (using in-memory state) but session records are queued for insertion when the database connection is restored.
- If the thermal printer becomes unresponsive, the backend attempts to reinitialize the printer connection up to 3 times before marking the printer as offline.
- If the webcam becomes unresponsive, the backend releases the camera resource and attempts to reinitialize it on the next session.
- All error recovery events are logged with timestamps, error details, and recovery actions taken.
- The admin dashboard displays the current system health status (all services green/yellow/red) and the history of recent error recovery events.
- The kiosk screen never displays a stack trace, error code, or technical error message to the end-user. All error states are presented as friendly, actionable messages.

**Priority:** P0
